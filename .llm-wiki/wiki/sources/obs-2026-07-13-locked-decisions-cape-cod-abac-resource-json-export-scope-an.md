# 🔴 Observation: Locked decisions: cape-cod ABAC resource JSON export scope and format
Locked design decisions for the ABAC Pulumi resource export (cape-cod side), from maintainer answers 2026-07-13:
(1) Scope: cape-cod ONLY generates a JSON export of deployed resources. All DB read/diff/migration/stale-entry handling/reconcile lives in a cape-cod-env "database update" python script (accepts local file OR s3:// path). cape-cod does NOT import cape-cod-db (avoids the database.py exit(1) landmine).
(2) Tributary config gains display_name + description (human readable); existing config `name` (e.g. hai) maps to DB tributary.code (short, drives AWS names); display_name -> tributary.name; description -> tributary.description; defaults to code when absent.
(3) Real config has 3 tributaries (hai, genomics, seqauto), each with 4 standard buckets input-raw/input-clean/result-raw/result-clean (physical names are AWS-autogenerated Outputs). Category+access mapping: input-raw->raw_uploads/write, input-clean->clean_uploads/read, result-raw->raw_results/write, result-clean->clean_results/read.
(4) Export per-resource: resource_type "s3" (confirm w/ Drew; stale notes say s3_bucket), resource_identifier = whole-bucket s3://<physical-bucket>/ (agreed q3, matches OPA usage), display_name human, attributes carry bucket/arn/category/bucket_role/environment/managed_by. `category` is the authoritative stable descriptor; `access_type` is a DERIVED default (raw->write, clean->read) kept for current-schema compat and forward-compatible with the coming per-user-permission rework.
(5) Generation: assemble inside pulumi.Output.all(...).apply(...) and pulumi.export("cape_resource_export", ...); MVP local file via `pulumi stack output cape_resource_export --json > cape-cod-export.json` (no file-write side effects in the program). Optional later: push to known S3 object.
(6) q4 = a cape-cod-db schema rework (per-user-per-resource permissions -> new association table, Resource becomes catalog). Open question for Drew: does Resource.access_type stay (nullable default) or get removed. Export is forward-compatible either way, so implementation can proceed in parallel.
(7) Investigate-later idea: build export by mining `pulumi stack export` state instead of the program. Feasible for identifiers/ARNs/URN-hierarchy but brittle and blind to display_name/description/category unless resources are tagged; needs backend+AWS creds. Generating from the program is preferred; tagging resources with code/category/display_name would enable state-mining later.
